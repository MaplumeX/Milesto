## Context

Milesto 当前已经在任务、项目、领域和项目分组上落地 `deleted_at` 软删除，并在查询层普遍以 `deleted_at IS NULL` 过滤活跃数据。现状能满足“删除后从主界面消失”，但缺少三个关键能力：

- 用户没有统一入口查看被软删除的内容。
- 用户无法恢复误删的任务或项目。
- 用户无法对软删除内容执行“永久删除”或“清空”。

这次变更是一个跨层改动：需要同时覆盖 Renderer 路由/导航、Preload API、DB Worker 动作、共享 schema，以及与现有同步模型的兼容性。现有同步实现基于“实体行长期存在 + 字段版本 + `deleted_at` 墓碑”传播状态，因此不能把“永久删除”简单实现为直接删行，否则会破坏多端收敛和冲突解决。

## Goals / Non-Goals

**Goals:**

- 提供一个顶层 `Trash` 页面，集中展示软删除的任务与项目根条目。
- 让项目删除具备稳定的树级语义：项目是唯一可见根条目，其下任务和分组被项目吸收。
- 让领域删除后的项目/直属任务能在废纸篓中继续恢复，而领域本身不进入废纸篓。
- 支持恢复单任务、恢复整项目树、永久删除单条、清空废纸篓。
- 在父级容器缺失时给出确定性的恢复回退规则。
- 保持与现有 sync/outbox 模型兼容，不引入“永久删除导致远端复活”问题。

**Non-Goals:**

- 不把 Area 作为废纸篓条目加入首期范围。
- 不把 Project Section 作为独立废纸篓条目加入首期范围。
- 不提供废纸篓搜索、筛选、自动过期或批量选择。
- 不在首期做真正的物理垃圾回收或数据库压缩。
- 不改变 Logbook 的“完成/恢复未完成”语义。

## Decisions

### Decision: 为“永久删除”引入 `purged_at`，而不是物理删行

任务、项目和项目分组新增 `purged_at` 墓碑字段。用户执行“永久删除”或“清空废纸篓”时，系统在事务内写入 `purged_at`，并继续保留原始实体行。

原因：

- 现有同步机制依赖实体行与字段版本长期存在，直接 `DELETE` 会让远端无法判断该实体是否被永久清除。
- `purged_at` 可以把“用户不可恢复/不可见”与“底层仍保留同步墓碑”分开。
- 这比新增一套 tombstone 表更简单，且更符合当前 schema。

备选方案：

- 直接物理删除：实现最简单，但与现有同步模型冲突，放弃。
- 独立 `trash_entries` / `purge_tombstones` 表：语义更显式，但会引入第二事实源，首期过重，放弃。

### Decision: 废纸篓使用独立 `trash` 域 API，而不是复用 `task.restore`

新增独立 IPC / `window.api.trash.*` 领域方法，负责：

- `list`
- `restoreTask`
- `restoreProject`
- `purgeTask`
- `purgeProject`
- `empty`

原因：

- 当前 `task.restore` 的语义是“从 Logbook 恢复未完成”，不是“从废纸篓恢复”。
- 项目当前没有对应的 `project.restore`，若继续塞进 `project.update` 会让语义混乱。
- `trash` 域更适合承载混合根条目读模型和页面动作。

备选方案：

- 复用 `task.restore` / `project.update`：会把完成恢复与软删除恢复混为一谈，放弃。

### Decision: 废纸篓根条目通过现有层级关系推导，不引入独立 `trash_entries`

废纸篓列表读模型由 SQL 查询直接推导：

- 项目根条目：`projects.deleted_at IS NOT NULL AND projects.purged_at IS NULL`
- 任务根条目：`tasks.deleted_at IS NOT NULL AND tasks.purged_at IS NULL`，且其所属项目当前不是未清除的软删除项目

这保证了：

- 删除项目后，项目是唯一可见根条目。
- 先删任务、后删项目时，任务会被项目吸收，不再单独出现。
- 删除领域后，领域本身不出现，但其下项目会作为项目根条目，直属任务会作为任务根条目。

原因：

- 当前数据模型已经有足够的父子关系字段，无需再维护一张额外的“根条目映射表”。
- 吸收规则本质上是查询选择规则，而不是额外持久化状态。

备选方案：

- 写入 `trash_entries` 记录根对象和后代：查询更直观，但需要额外维护删除批次与后代快照，首期不值得。

### Decision: 项目恢复按“整棵树恢复”，单任务恢复按“父级缺失回 Inbox”

恢复项目时：

- 恢复项目本身。
- 恢复该项目下被同次软删除吸收的任务与分组。
- 保留任务自身的 `open/done` 状态，不把所有任务强制改回未完成。
- 若原 `area_id` 对应的 Area 已不存在或仍处于删除态，则项目恢复到无归属分组；项目下任务的 `area_id` 一并清空。

恢复单任务时：

- 若原父项目仍存在且可用，则回到原项目/分组。
- 若原父项目缺失，或原直属 Area 缺失，则统一回 Inbox。
- 由于当前任务不变量要求 Inbox 任务不能保留项目/领域归属与 `scheduled_at`，恢复到 Inbox 时需要清理与 Inbox 冲突的容器/计划字段。

原因：

- 这与用户已确认的规则一致，并且在现有任务约束下可落地。

备选方案：

- 父级缺失时禁止恢复：过于强硬，放弃。
- 父级缺失时恢复到“无归属普通任务”：能保留更多字段，但与用户选择冲突，放弃。

### Decision: 废纸篓页面使用独立混合列表读模型

新增共享 schema `TrashEntry`，以判别联合类型表示：

- `task` 条目
- `project` 条目

项目条目额外携带：

- `open_task_count`

Renderer 以独立 `TrashPage` + `TrashList` 实现，不把废纸篓硬塞进现有 `TaskList`。

原因：

- 现有 `TaskList` 假设同构任务行，不适合承载项目根条目。
- 废纸篓页面没有拖拽排序，也不需要沿用 `TaskList` 的 list-position 行为。

## Risks / Trade-offs

- [恢复到 Inbox 会清理部分计划/容器字段] -> 通过设计文档与测试明确该回退语义，仅在父级缺失时触发。
- [用 `purged_at` 代替物理删除会保留历史行] -> 首期接受“逻辑永久删除”，未来如有需要再引入后台 GC。
- [基于 SQL 推导根条目会让查询变复杂] -> 限定首期只支持任务/项目两种条目，并为吸收规则添加专门测试。
- [Area 不进入废纸篓，可能让用户困惑为何恢复项目后 Area 不回来] -> 恢复时明确回退到无归属分组，并在 UI 文案中避免承诺恢复 Area。

## Migration Plan

1. 为 `tasks`、`projects`、`project_sections` 增加 `purged_at` 字段，默认 `NULL`。
2. 新增废纸篓查询与动作 API，并让所有活跃面查询统一过滤 `purged_at IS NULL`。
3. 新增 `/trash` 路由与导航入口，接入页面读模型。
4. 补充恢复/永久删除/清空废纸篓测试，覆盖项目吸收、Area 删除回退、Inbox 回退等关键路径。
5. 若需要回滚，UI 路由与动作可直接下线；新增字段保留但不再使用，不影响旧数据读取。
