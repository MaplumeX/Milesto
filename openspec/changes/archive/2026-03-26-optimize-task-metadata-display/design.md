## Context

当前 `TaskRow` 折叠态只显示 checkbox、标题和项目归属，缺少计划时间、到期时间与标签预览，用户需要频繁展开才能确认任务编排信息。行内编辑器虽然已经支持 schedule/due/tags picker，但 metadata 分布在底部摘要 chips 与底部操作按钮中，导致阅读顺序与编辑入口分离。

这个变更会同时影响：
- 折叠态任务行的布局与截断策略
- 行内编辑器的信息层级与交互入口
- 任务列表/搜索结果的共享数据契约
- 虚拟滚动列表的尺寸估算

项目约束也比较明确：
- 必须遵守低噪声、极简层级，不引入额外重装饰
- 10k 列表场景下不能通过逐行详情请求补齐标签数据
- Renderer 只能消费已有业务级数据边界，不能在 UI 层临时拼高成本数据流

## Goals / Non-Goals

**Goals:**
- 在折叠态直接暴露计划时间、到期时间和标签预览，提升列表扫描效率
- 让展开态变成 `标题 -> 信息 -> 内容 -> 操作` 的稳定层级
- 让 metadata 本身成为编辑入口，减少重复按钮和视觉噪声
- 通过轻量预览字段支撑折叠态渲染，避免按行读取详情

**Non-Goals:**
- 不重做 overlay editor
- 不引入新的数据表或新的 tag 排序模型
- 不修改 task context menu 的 metadata 编辑能力
- 不在本次变更里扩展新的搜索交互，仅对齐共享数据契约

## Decisions

### 1. 为列表层引入轻量 metadata 预览契约

决定：在任务列表返回模型里新增轻量 tag preview 字段，例如 `tag_preview` 与 `tag_count`，用于折叠态渲染。

原因：折叠态需要标签名称与数量，但直接复用 `task.getDetail` 会导致每一行都走额外请求，在大列表下不可接受。相比返回完整 tag 对象数组，轻量预览字段更符合列表页“只取必要字段”的约束。

备选方案：
- 按行调用 detail 接口：实现简单，但性能风险不可接受
- 在列表接口里返回完整 tags：数据量更大，超出当前 UI 所需

### 2. 折叠态采用单行、右对齐 metadata 簇，并优先保留 metadata

决定：折叠态任务行保持单行结构，左侧为 checkbox + 标题，右侧为 metadata 簇；当空间不足时优先截断标题，不让 metadata 换行。

原因：用户已经明确选择 `B3`，即把 metadata 视为主信息而不是标题的补充信息。单行结构也最有利于维持列表节奏和虚拟滚动稳定性。

备选方案：
- 双行 metadata rail：标题保留更多，但不符合已确认方向
- 平衡压缩：标题和 metadata 一起退让，但弱化了 metadata 优先级

### 3. 展开态把 metadata 下沉为标题下方的信息带

决定：行内编辑器在标题输入框下方渲染独立 metadata 信息带，notes 和 checklist 继续下沉到内容区。

原因：这能把展开态层级收束成 `标题 -> 信息 -> 内容 -> 操作`，比“顶栏右侧 metadata”更清晰，也符合已确认的视觉选择。

备选方案：
- 顶栏右侧继续保留 metadata：和折叠态延续性强，但展开后拥挤
- 继续保留底部摘要 chips：信息位置过低，和标题脱节

### 4. 信息带 chip 复用现有 picker 逻辑，删除重复 footer metadata 按钮

决定：信息带中的计划时间、到期时间、标签 chip 直接复用现有 picker/open state 和持久化逻辑；底部重复的 `Schedule` / `Due` / `Tags` 按钮移除，仅保留 Checklist 与其他非重复动作。

原因：现有交互已经具备 picker 与保存语义，变更入口位置比重写一套更稳，也符合 KISS。删除重复入口可以降低展开态噪声。

备选方案：
- 新建一套 info band 专用交互：重复实现，没有必要
- 保留旧 footer 按钮作为备用入口：会制造重复与竞争

### 5. 虚拟列表保持测量机制，同时校准估算高度

决定：继续依赖现有 `measureElement` 处理动态高度，同时按最终折叠态高度统一调整各任务列表的 `estimateSize`。

原因：当前虚拟列表已经支持动态测量，但初始估算仍基于旧高度。如果视觉高度变化，不同步修正估算值会增加首帧跳动。

备选方案：
- 完全依赖运行时测量：可行，但初始跳动会更明显
- 保持旧估算值不动：容易在高频列表切换中出现视觉抖动

## Risks / Trade-offs

- [标题更早截断] → 通过固定 metadata 顺序、短标签文本与 `+N` 规则控制占用宽度
- [列表查询聚合标签预览增加复杂度] → 只返回轻量预览字段，并在单次查询内完成聚合，避免 N+1
- [展开态入口位置变化影响旧习惯] → 保持 picker 行为不变，只迁移入口位置
- [虚拟列表首帧跳动] → 在所有任务列表统一校准 `estimateSize`，并保留现有 `measureElement`

## Migration Plan

1. 先扩展任务列表/搜索结果的轻量 metadata 预览字段，并保持旧 UI 仍可消费
2. 更新折叠态任务行渲染与样式，验证各任务列表的布局和截断行为
3. 将行内编辑器 metadata 移到标题下方的信息带，复用现有 picker 逻辑
4. 移除底部重复 metadata 入口，仅保留 Checklist 与非重复动作
5. 统一校准相关虚拟列表的高度估算并回归主要任务列表

回滚策略：
- 回退到旧的任务列表 schema 字段集
- 恢复旧的 footer summary chips / metadata buttons 布局
- 将折叠态任务行恢复为仅标题结构

## Open Questions

- 当前无阻塞性开放问题；如果后续需要区分搜索结果是否也显示完整 metadata 簇，可在实现阶段按共享 renderer 的实际复用情况决定。
