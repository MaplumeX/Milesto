# 夜间模式配色调整设计（Things3 方向）

日期：2026-03-16
状态：已确认，待进入实现计划

## 1. 背景

当前夜间模式整体偏冷、偏黑，侧边栏与内容区的分区关系不够明确，输入框、卡片、底部栏等表面层也没有形成统一的深色层级。

这次调整目标是让夜间模式更接近 Things3 的气质，但不复制其视觉细节。重点只放在配色层级，而不是重做布局或组件结构。

## 2. 已确认的设计结论

用户已确认以下方向：

- 主方向：暖石墨（Things3 气质）
- 分区关系：侧边栏和内容区保持中等差值
- 影响范围：除侧边栏和内容区外，连同输入框、卡片、底部栏、popover 等面板色一起统一

最终采用方案：平衡版 Things 感

- `sidebar`：`#181A1D`
- `content`：`#222528`
- `surface`：`#2A2D31`
- `text`：`#F0E9DC`
- `muted text`：`#B4AB9D`
- `divider`：`rgba(241, 233, 219, 0.08)`
- `hover wash`：`rgba(255, 255, 255, 0.035)`
- `active wash`：`rgba(255, 255, 255, 0.08)`

## 3. 非目标

本次不做以下变更：

- 不修改布局结构
- 不修改浅色主题
- 不重做品牌色和语义状态色体系
- 不在页面组件中引入新的局部硬编码颜色

## 4. 设计原则

### 4.1 KISS

只做主题 token 和少量布局级 token 收口，不改组件结构，不新增主题切换机制。

### 4.2 DRY

统一从 `src/index.css` 的暗色 token 出发映射表面层，避免在 `.sidebar`、`.content`、`.card`、`.input` 等规则里重复写死颜色。

### 4.3 YAGNI

不预留多套暗色变体，不额外引入主题对象、CSS-in-JS 或新的主题依赖。

### 4.4 SOLID

把“布局层背景差异”和“通用表面层语义”分开：

- 布局级 token 只负责 `sidebar / content / edge`
- 语义级 token 继续负责 `panel / glass / paper / wash / border / text`

这样可以在不改组件消费方式的前提下，独立调整页面结构层级和通用组件层级。

## 5. 方案对比

评估过的三个方向：

### 5.1 方案 A：保守迁移

优点：

- 改动风险最低
- 对现有界面冲击最小

缺点：

- Things3 气质提升有限
- 内容区和面板层级仍然偏保守

### 5.2 方案 B：平衡版 Things 感（采用）

优点：

- 侧边栏更沉，内容区更开阔，层级清楚
- 面板层只比内容区抬半层，整体更安静
- 既能贴近目标风格，又不需要重构样式系统

缺点：

- 需要同时校准布局背景和表面层 token

### 5.3 方案 C：更柔和的纸感面板

优点：

- 表面层更精致
- 卡片和输入框存在感更强

缺点：

- 容易显厚、显灰
- 对现有极简基调侵入更强

## 6. 实现设计

### 6.1 文件范围

本次实现应集中在以下文件：

- `src/index.css`
- `electron/main.ts`

原则上不修改页面组件结构，不把颜色散落到 `src/pages/*` 或 `src/features/*`。

### 6.2 CSS token 设计

在暗色主题下，现有 token 需要调整为暖石墨体系，并补足布局级 token。

建议引入或重定义的重点 token：

- 布局级
  - `--sidebar-bg`
  - `--content-bg`
  - `--content-edge-bg`
- 语义级
  - `--panel`
  - `--glass`
  - `--glass-soft`
  - `--glass-strong`
  - `--paper`
  - `--text`
  - `--muted`
  - `--border`
  - `--wash`
  - `--wash-strong`

其中：

- `--sidebar-bg` 明确提供侧边栏基底，替代当前“主底色 + 半透明玻璃”的模糊层次
- `--content-bg` 提供内容区稳定基底
- `--content-edge-bg` 用于内容底部栏或边缘层，使其与内容区同体系但再压半层

### 6.3 布局应用方式

#### 侧边栏

`.sidebar` 应直接使用 `--sidebar-bg` 作为主体背景。保留必要的边界感，但降低当前过于“玻璃感”的存在。

#### 内容区

`.content` 应显式设置 `background: var(--content-bg)`，不再完全依赖 `body` 背景。

#### 内容底部栏

`.content-bottom-bar` 应使用 `--content-edge-bg`，和内容区保持同一体系，并维持轻微分层。

#### 详情面板 / 卡片 / Popover / 输入框

这些表面层继续消费通用语义 token，不单独写死颜色。目标是让它们统一落在 `surface` 一层，而不是各自偏冷或偏透明。

### 6.4 Electron 窗口背景

`electron/main.ts` 里目前深色窗口背景是单独写死的冷黑值。

这会导致两个问题：

- 应用启动时背景先出现旧色
- 主题切换时可能出现窗口底色与 CSS 内容区底色不一致

因此需要把深色窗口背景同步调整为内容区基底色，使 BrowserWindow 的 `backgroundColor` 与 Renderer 的暗色内容区一致。

## 7. 数据流与边界

这次变更不涉及新的数据流，不新增 IPC，不修改 `window.api` 边界，也不触碰 DB Worker。

主题切换仍然沿用当前机制：

- Renderer 设置主题偏好
- Main 计算 `effectiveTheme`
- CSS 通过深浅主题规则切换视觉表现

本次只是让 Main 的窗口背景色和 Renderer 的暗色 token 保持一致。

## 8. 错误处理

本次配色改动不引入新的业务错误类型。

需要关注的失败形式只有两类：

- 样式回归：局部组件在暗色下层级不清、对比不足或 hover/active 失真
- 主题切换回归：窗口背景与内容区不一致，出现闪色

这些问题通过人工验证和类型检查拦截，不需要新增错误结构。

## 9. 验证方案

### 9.1 必做验证

- 运行 `npx tsc -p tsconfig.json`
- 手动切换 `light -> dark -> system`
- 检查设置页、主内容页、侧边栏、底部栏、输入框、卡片、popover/详情面板
- 检查启动和主题切换时是否存在窗口底色闪烁

### 9.2 说明

当前仓库 `npm run lint` 没有可用配置，不作为本次门禁。

### 9.3 回归判断标准

满足以下条件即可认为配色调整达到目标：

- 夜间模式整体从冷黑转为暖石墨
- 侧边栏与内容区存在明确但柔和的分区
- 输入框、卡片、底部栏和浮层进入同一套暗色层级
- hover、active、divider 清楚但不过分跳
- 启动和主题切换时不出现旧深色背景闪出

## 10. 风险与回退

主要风险：

- 色温过暖，导致界面泛黄
- 面板抬升过强，导致界面显厚
- 分区差值不足，导致侧边栏和内容区重新糊在一起

回退成本较低，因为改动会集中在 token 和窗口背景色：

- 回调暗色 token 即可
- 不涉及组件结构回退
- 不涉及跨层架构回退
